# 代码修改与审查守卫

本技能是一个面向代码变更的守卫技能，用于在实现、重构和审查过程中守住正确性、结构质量、类型安全与审查闭环。
本技能不用于纯探索、纯讨论或无代码变更的场景，也不将项目私有约束伪装为通用规则。

## 适用边界

### 适用场景

- 修改代码、修复 bug、重构、新增脚本。
- 代码审查、质量回看、变更后复核。
- TypeScript 高级类型设计，如泛型、条件类型、映射类型、模板字面量类型、工具类型、`infer`、类型守卫。

### 谨慎适用场景

- 遗留代码修补：优先保证不引入新的同类高风险问题。
- 快速修复或上线前补丁：允许记录技术债，但必须披露剩余风险。
- 跨模块或多文件结构调整：按逻辑单元分批审查，避免一次性扩散。

### 不适用场景

- 纯探索、架构脑暴、只读调研。
- 仅修改文档、配置或不涉及代码行为的小型元数据调整。
- 第三方依赖目录、生成代码目录、明确以试验为目的的 PoC / spike（除非用户明确要求按正式质量标准执行）。

## 工作原则

- 先保证正确性，再优化结构质量。
- 规则分层执行，避免不同性质的问题采用同一强度处理。
- language profile 与 project profile 按需加载，不默认冒充通用规则。
- 例外用于解释偏离规则的原因，不用于取消风险披露。

## 通用规则分层

规则分层的目标不是弱化约束，而是让不同性质的约束以合适的强度生效。
通用底线应稳定执行，结构性问题应结合复杂度和风险判断，团队偏好或项目私有约束不应伪装为通用规则。

### 硬规则

默认必须满足，通常应优先修复。
若仓库存在历史基线问题，至少不得引入新的同类高风险问题。

| 规则 | 名称 | 说明 |
|------|------|------|
| R0 | 构建正确性 | 变更不得引入新的编译错误、构建失败或新的高优先级告警。 |
| R5 | 类型安全 | 使用语言提供的严格类型能力；禁止无说明的类型逃逸。 |
| R12 | 中文注释规范 | 公开面必须提供中文注释，复杂内部逻辑按条件补充说明。 |

### 条件硬规则

默认检查；当改动复杂度、风险或规模达到阈值时，升级为必须满足。

| 规则 | 名称 | 说明 |
|------|------|------|
| R1 | 单一职责 | 类、函数承担的职责不应继续膨胀。 |
| R3 | 嵌套深度 | 高复杂分支应通过 guard clause、提取函数等方式收敛。 |
| R4 | 参数数量 | 参数过多时应考虑参数对象、拆分职责或封装上下文。 |
| R6 | 类型/泛型体量 | 复杂类型定义应在可读性和收益之间取得平衡。 |
| R8 | 安全收窄 | 优先类型守卫、模式匹配或显式校验，避免无说明的强制断言。 |
| R9 | 检查强度 | 仓库若支持更严格的编译/Lint 检查，应优先启用；遗留基线不一次强推。 |
| R13 | 魔法值控制 | 关键业务值、状态码、资源键、文案等应有清晰语义来源。 |
| R14 | 最小权限 | 非必要不扩大可见性，公共暴露需有明确理由。 |

典型升级场景包括：

- 公共 API 变更、跨模块改动或基础设施层改动。
- 多文件重构、大型重排、复杂业务状态流转。
- 运行时热点路径、序列化边界、第三方输入边界。

### 软建议

默认纳入审查，但通常不单独阻断变更。

| 规则 | 名称 | 说明 |
|------|------|------|
| R2 | 文件体量 | 超长文件通常是结构问题信号，但不等同于必然违规。 |
| R7 | DRY | 避免无收益的重复，同时避免过早抽象。 |
| R11 | 命名规范 | 命名应自解释、一致、避免歧义；具体风格按语言 profile 落地。 |

### 下沉规则

不属于通用层，只在 language profile 或 project profile 中生效。

| 规则 | 名称 | 说明 |
|------|------|------|
| R15 | 一文件一类 | 属于语言/团队约束，不作为所有语言的通用硬规则。 |
| P-CSharp | C# / Unity 专项约束 | 通过 `references/csharp-conventions.md` 加载。 |
| P-TS | TypeScript 高级类型专项 | 通过 `references/typescript-advanced-types.md` 加载。 |
| P-Project | 项目私有约束 | 仅在仓库明确采用对应架构、热更方案或私有框架时启用。 |

## 规则定义

### R0 构建正确性

- 默认必须满足。
- 变更不得引入新的编译错误、构建失败或新的高优先级告警。
- 若仓库基线本身不干净，至少不得扩大问题范围，并优先修复本次改动直接导致的问题。

### R1 单一职责

- 默认检查；当类、函数或模块承担的职责明显膨胀时升级为必须满足。
- 典型信号：同时处理业务编排、状态更新、IO、格式转换、异常补偿等多类逻辑。

### R2 文件体量

- 默认提示，通常不单独阻断。
- 单文件超过 500 行或接近 300 行时，应审视是否存在职责拆分空间。

### R3 嵌套深度

- 默认检查；复杂分支、嵌套超过 3 层时通常应收敛。
- 优先使用 guard clause、状态拆分、策略提取或小函数分解。

### R4 参数数量

- 默认检查；参数超过 5 个时应解释理由或重构为参数对象、上下文对象、职责拆分。

### R5 类型安全

- 默认必须满足。
- 禁止无说明的类型逃逸，如 TS `any`、C# `dynamic`、Java raw type 等。
- 若位于边界层、第三方适配层或框架约束位置，允许例外，但必须说明原因和风险。

### R6 类型/泛型体量

- 默认检查；当类型定义复杂到难以理解其真实收益时，升级为必须收敛。
- 优先拆分中间类型别名，保留关键语义，避免类型定义比业务代码更难读。

### R7 DRY

- 默认提示。
- 既要避免机械复制，也要避免为了抽象而抽象。

### R8 安全收窄

- 默认检查；在外部输入边界、反序列化、运行时判型、复杂联合类型等场景下升级为必须满足。
- 若使用断言或强制转换，应优先提供运行时校验或明确说明其前提。

### R9 检查强度

- 默认检查。
- 仓库若已支持更严格的编译/Lint 配置，应优先启用。
- 对遗留仓库不要求一次性推到最严格，但应避免在新改动中继续放宽。

### R11 命名规范

- 默认提示。
- 命名应自解释、一致、避免歧义和无信息量缩写。
- 具体风格由 language profile 细化，例如 C# 的 PascalCase / camelCase 规范。

### R12 中文注释规范

R12 采用“硬规则 + 条件硬规则 + 软建议”三级约束，目标是提高业务语义、边界条件和设计意图的可理解性，而不是机械重复代码字面行为。

#### 1. 硬规则

以下对象必须具备中文注释，缺失即判定为违规：

- 类、接口：必须有中文文档注释，说明职责、用途和边界。
- 公开字段、公开属性：必须有中文注释，说明含义、用途或使用场景。
- 公开方法：必须有中文文档注释，至少说明：
  - 方法用途
  - 关键参数含义
  - 返回值含义（如有）
  - 重要副作用、前置条件或限制（如有）

#### 2. 条件硬规则

私有方法或内部方法在“逻辑复杂”时，必须补充中文注释；缺失则判定为违规。

“逻辑复杂”通常指满足以下任一情况：

- 包含较多分支判断或多层嵌套
- 存在状态流转或阶段切换
- 包含业务规则、边界条件或特殊判定
- 包含性能优化、兼容性处理或历史约束
- 仅凭命名无法直接理解真实意图

这类注释应优先说明：

- 这段逻辑在解决什么问题
- 为什么这样写，而不是另一种更直观的写法
- 哪些边界或约束不能随意改动

#### 3. 软建议

以下情况不强制要求详细注释，但可按可读性酌情补充：

- 直白的 getter / setter
- 无业务含义的一行转发函数
- 命名已足够清晰、行为完全显然的简单私有方法

目标是减少无信息量的注释，避免出现“设置变量”“处理数据”这类空泛描述。

#### 4. 注释语言要求

- 注释必须使用中文。
- 注释优先说明“做什么”“为什么这样做”，而不是机械复述代码字面行为。
- 文档注释是形式要求，中文注释规范是内容要求；C# 场景通常采用 `///` XML 文档注释承载该要求。

#### 5. 审查判定建议

- 类、接口、公开字段/属性、公开方法缺少中文注释：中危违规。
- 公开方法未说明关键参数或返回值含义：中危违规。
- 复杂私有/内部方法缺少关键中文说明：中低危违规。
- 简单私有方法未写注释：通常不判违规，可作为优化建议提出。

### R13 魔法值控制

- 默认检查；关键业务 ID、状态码、资源路径、协议值、可变文案等应有语义来源。
- 局部显然值（如 `0`、`1`、数组起始索引）可保留，但不应掩盖业务语义。

### R14 最小权限

- 默认检查；当变更涉及公共 API、管理器、基础设施层或跨模块暴露时升级为必须满足。
- 若扩大可见性，应明确其调用面与稳定性要求。

### R15 一文件一类

- 该规则默认不进入通用规则层。
- 若对应语言/团队规范明确要求一文件一类，则在 language profile 或 project profile 中启用。

## 变更流程

### 变更前

1. 判断是否属于本技能适用场景。
2. 评估修改范围、风险级别和可能触碰的条件硬规则。
3. 判断是否需要加载 language profile 或 project profile。
4. 若预计触碰大规模结构性问题，提前规划拆分方案和审查节点。

### 变更中

- 先守住 R0、R5、R12 这类高收益底线。
- 对条件硬规则按复杂度、风险和规模判断是否升级为必须满足。
- 多文件改动按逻辑单元或文件批次插入中间审查节点。
- 若需要例外，及时记录原因、适用边界和剩余风险，不等到结尾再补。

### 变更后

按以下顺序检查：

1. 先检查构建正确性（R0）。
2. 再检查类型安全、中文注释规范等高优先级规则（R5、R12）。
3. 再检查条件硬规则是否在当前复杂度下满足。
4. 最后给出软建议、剩余风险和后续重构建议。

## 审查机制

原 `R10` 提升为审查流程要求，不再与质量规则混排。

### 审查粒度

| 变更规模 | 审查策略 | 说明 |
|---------|---------|------|
| 单文件 / 极小改动 | 完成后一次性审查 | 传统模式 |
| 2–5 个文件 | 每完成一个文件或一个小逻辑单元后立即审查 | 及早发现问题，避免返工扩散 |
| 6+ 个文件（大型重构） | 按逻辑单元分批审查 + 全部完成后集成审查 | 逻辑单元 = 一组功能内聚的文件 |

### 迭代修复

```
审查 → 发现违规？
  ├─ 否 → ✅ 通过，结束
  └─ 是 → 修复违规项 → 重新审查（迭代 +1）
           └─ 迭代次数 ≥ 3？
               ├─ 否 → 继续循环
               └─ 是 → ⚠️ 终止循环，输出剩余违规报告 + 人工介入建议
```

- 最大迭代次数：3 轮。
- 每轮重审只检查上一轮违规涉及的文件，不重复扫描无变更文件。
- 迭代过程中产生的新违规计入当轮，不另起循环。

### 熔断机制

在任意审查节点，若满足以下任一条件，立即停止继续扩散风险：

| 熔断条件 | 触发阈值 | 处理方式 |
|---------|---------|----------|
| 技术阻断 | R0 不通过 | 停止后续变更，优先修复构建/编译问题 |
| 高危违规累积 | 同一审查节点高危问题 ≥ 3 项 | 停止后续变更，输出熔断报告 |
| 结构阻断 | R1 与 R2 同时严重违规，且已影响继续实现 | 要求先完成拆分重构再继续 |

熔断后输出格式：

```
## 🛑 熔断终止

**触发条件**：[具体熔断原因]
**已完成变更**：[已完成的文件/批次列表]
**未完成变更**：[剩余计划]
**违规摘要**：[当前所有违规项]
**建议**：[下一步行动方案]
```

### 终止条件

| 场景 | 终止行为 |
|------|----------|
| 所有检查通过 | `✅ 未发现违规，所有检查通过。` |
| 发现违规且迭代未超限 | 修复后重审（最多 3 轮） |
| 迭代 3 轮后仍有违规 | 输出剩余违规报告，建议人工介入 |
| 高危熔断触发 | 立即终止，输出熔断报告 |
| 极小改动 | 可简化为一行结论 |
| 用户明确要求跳过审查 | 解释强制要求；用户仍坚持时终止本技能流程并披露风险 |

## 输出规范

标准输出格式：

```
## 审查结论

### 违规项（按严重程度排序）
1. 🔴 [高危] ...
2. 🟡 [中危] ...
3. 🔵 [低危] ...

### 质量检查
| 规则 | 状态 | 说明 |
|------|------|------|
| R0 构建正确性 | ✅/❌ | ... |
| R5 类型安全 | ✅/❌ | ... |
| R12 中文注释规范 | ✅/❌ | ... |
| ... | ... | ... |

### 剩余风险（如有）
- ...

### 重构建议（如有）
- ...
```

若无违规则输出：`✅ 未发现违规，所有检查通过。`

## 例外与豁免

例外用于解释和记录偏离规则的原因，不用于取消风险披露。
任何被接受的例外都应明确剩余风险、适用边界和后续建议，避免“默认放行”。

### 允许例外但需披露风险的场景

- 遗留仓库基线不干净。
- 边界层、第三方适配层、框架约束位置需要受控类型断言。
- 紧急修复需要优先保正确性，结构优化延后。

### 默认豁免的范围

- 第三方依赖目录。
- 生成代码目录。
- 不属于当前变更责任面的外部代码。

### 处理要求

- 允许不阻断，但必须说明原因。
- 必须披露剩余风险、适用边界和后续建议。
- 不得将例外描述为“已无风险”或“无需审查”。

## Profiles

### TypeScript Profile

当 TypeScript 项目涉及泛型、条件类型、映射类型、模板字面量类型、工具类型、`infer`、类型守卫或 `this` 指向问题时，读取：

- `references/typescript-advanced-types.md`

### C# / Unity Profile

当项目为 C# / Unity 时，按需读取：

- `references/csharp-conventions.md`

该 profile 内部继续区分：

- Core：通用 C# 建议
- Unity Runtime：运行时代码条件规则
- Project Profile：项目私有约束

### Project Profiles

项目私有约束仅在仓库明确采用对应架构、热更方案或私有框架时启用。
若无明确前提，不得将其上升为通用规则。
