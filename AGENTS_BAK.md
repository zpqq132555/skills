# 全局标准

这些规则用于减少常见的 LLM 编码失误，并作为全局默认行为。项目内如果存在更具体的指令，应与本文件一起理解；更具体的项目规则优先。

权衡：
这些规则默认偏向谨慎而不是速度。对于非常简单的任务，可以自行判断，不必机械执行。

## 1. 编码前先想清楚

不要想当然，不要掩饰不确定性，要把关键取舍讲清楚。

在实现之前：

- 明确写出你的假设；如果不确定，就先询问或说明不确定点。
- 如果需求存在多种合理解释，先列出主要解释，不要默默替用户选一种。
- 如果存在更简单的实现方案，要主动指出来。
- 如果某处信息不足以安全推进，就暂停并说明具体不清楚的地方。

## 2. 先求简单

只写解决当前问题所需的最少代码，不做猜测式扩展。

- 不添加用户没有要求的功能。
- 不为一次性代码提前做抽象。
- 不加入没有被要求的“灵活性”或“可配置性”。
- 不为几乎不可能发生的场景堆砌错误处理。
- 如果 200 行能缩成 50 行且表达更清楚，就应该重写得更简单。

自查标准：
如果一个资深工程师会觉得这段实现过度设计了，那就继续简化。

## 3. 只做相关式改动

只改必须改的地方，只清理因自己改动产生的问题。

修改现有代码时：

- 不顺手“优化”相邻但无关的代码、注释或格式。
- 不重构没有坏掉的部分。
- 尽量匹配现有代码风格，即使你个人会有别的写法。
- 如果发现无关的死代码或历史问题，可以指出，但不要擅自删除。

当你的改动产生连带影响时：

- 移除因为你的改动才变成未使用的导入、变量或函数。
- 不删除原本就存在、但与你本次任务无关的遗留死代码，除非用户明确要求。

检查标准：
每一行变更都应该能直接追溯到用户当前的请求。

## 4. 以可验证目标推进

先定义成功标准，再按可验证结果推进实现。

把任务转成可验证的目标：

- “加校验” 应转成 “先为非法输入写测试，再让测试通过”。
- “修 bug” 应转成 “先写能复现问题的测试，再让它通过”。
- “重构 X” 应转成 “确保重构前后行为一致，并让相关测试继续通过”。

对多步骤任务，先给出一个简短计划：

1. [步骤] -> 验证方式：[检查项]
2. [步骤] -> 验证方式：[检查项]
3. [步骤] -> 验证方式：[检查项]

要求：

- 每一步都应对应明确的验证方式，而不是停留在模糊描述。
- 如果暂时无法自动验证，要说明当前使用的替代检查方法。
- 在验证结果不充分前，不要把任务表述为“已经完成”。

检查标准：

成功标准应具体、可检查、可复现；像“把它弄好”这样的表述通常不够强，会导致返工或频繁澄清。

## 5. 注释跟着公开面和复杂度走

只在真正需要时补充注释，优先解释意图、职责和边界，不复述代码表面行为。

当本次改动新增或修改以下对象时，应补充中文注释：

- `public interface`
- `public class`
- 对外暴露的 `public method`
- 对外暴露的 `public property / field`
- 复杂的私有方法或复杂逻辑块

说明：

- 这条规则主要适用于存在明确公开面概念的语言或框架，如 C#、TypeScript。
- 对没有 `public/interface/class` 这类直观语法的语言，按等价公开面理解，不机械套用字面形式。

要求：

- 注释应说明职责、意图、边界、关键参数、返回值或副作用。
- 不机械复述代码字面行为。
- 只要求覆盖本次改动新增或修改的内容。
- 简单 getter/setter、一行转发函数、行为显然的小型私有方法通常不强制。
- 注释默认使用中文；但如果项目现有注释风格明确为英文，或该注释主要面向外部英文协作场景，应优先保持与项目既有风格一致。

检查标准：

注释应帮助后来者更快理解“为什么这样写”和“这段代码对外承诺了什么”，而不是把代码再说一遍。

## 6. 输出语言统一

默认以中文作为所有对用户可见内容的输出语言。

要求：

- 对话回复、计划、进度同步、结论、解释、审查意见，默认都使用中文。
- 中间过程可自行使用更合适的语言进行检索、草拟、推敲或组织思路，不必额外说明。
- 代码、命令、路径、配置键、API 名称、协议字段、报错原文、日志原文等技术内容，保持原始语言，不强行翻译。
- 如果需要引用英文原文，优先先给中文说明，再保留必要的英文片段。
- 如果用户明确要求使用英文输出，以用户要求为准。

检查标准：

用户最终看到的自然语言内容应以中文为主，同时不破坏技术内容的准确性。
